# 設計・コスト検証ドキュメント (Design & Cost Verification)

本プロジェクトにおける設計判断と、それに関連するコスト（開発効率、リソース消費、メンテナンス性）の検証結果をまとめます。

## 1. 設計の検証 (Design Verification)

### アーキテクチャの妥当性

- **選定内容**: Clean Architecture ベースの 3 レイヤー構成。
- **検証結果**:
    - **テスト容易性**: Domain 層を純粋な Kotlin コードに分離したことで、Android
      フレームワークに依存しない高速なユニットテストが可能になった。
    - **AI 親和性**: 各レイヤーの責務が明確なため、AI エージェントへの指示（Prompt）が具体化しやすく、コード生成の正確性が向上した。
    - **柔軟性**: Ktor の `MockEngine` を活用することで、実際のサーバーが未完成の状態でも通信層のテストを完結でき、開発のブロッキングを最小化できた。

### 技術スタックの検証

- **Ktor vs Retrofit**:
    - Kotlin Multiplatform (KMP) を見据えた Ktor を採用。コルーチンとの親和性が高く、`MockEngine`
      によるテスト構成が非常に軽量である。
- **OpenAPI Generator**:
    - DTO を自動生成することで、API 定義の変更に対する追従コストを大幅に削減。手動実装によるタイポや型ミスのリスクを排除した。

## 2. コストの分析 (Cost Analysis)

### 開発コスト (Development Effort)

- **AI エージェントの活用**:
    - ボイラープレートコードを AI に委ねることで、初期の実装時間を約 30-40% 削減。
    - `docs/` 配下のドキュメントとコードの整合性を AI が維持するコストが低減した。

### 運用・リソースコスト (Resource & Operational Cost)

- **API 通信コスト**:
    - `MockEngine` を活用することで、開発中の GitHub API レートリミット制限（60回/時）を回避し、クォータ消費を最小化。
- **アプリサイズ/メモリ**:
    - リリースビルド時の最適化により、ライブラリ追加によるオーバーヘッドを許容範囲内に収めている。

## 3. 最小構成・人力テストとの比較 (Comparison with Minimalist Approach)

### 比較対象の定義 (Minimalist Approach)

- **構成**: 新規アプリテンプレートの Activity/MainScreen に全ロジックを記述（Fat UI）。
- **実装**: `lifecycleScope` で直接 Ktor 呼び出し、手書き DTO、自動テストなし。
- **テスト**: 開発者が実機・エミュレータで目視確認（人力）。

### 検証結果

| 比較項目        | 最小構成 ＋ 人力テスト  | 本プロジェクト (構造化 ＋ 自動テスト)  | 備考                      |
|:------------|:--------------|:-----------------------|:------------------------|
| **初期構築コスト** | **極めて低い**     | **高い** (DI, レイヤー分離の導入) | 最初の 1 画面を作る速度は最小構成が圧倒的。 |
| **機能追加の速さ** | **次第に低下**     | **維持・向上**              | 密結合は AI の生成精度も低下させる。    |
| **テストの信頼性** | **低い** (確認漏れ) | **高い** (CI 全件自動実行)     | 人力では回帰テストが次第に不可能になる。    |
| **デバッグ効率**  | **低い** (実機必須) | **高い** (JUnit で即座に確認)  | 通信の境界値テストは自動テストが速い。     |

## 4. コスト逆転の閾値分析 (Cost Inversion Threshold)

現在の「リポジトリ一覧を表示するだけ」の規模では、最小構成の方が低コストですが、どの時点で本プロジェクトの構造が有利になるか（コストが逆転するか）を算出します。

### 閾値の判定基準

以下のいずれかの条件を満たしたとき、本プロジェクトの構造がトータルコスト（開発＋修正＋テスト）で有利になります。

| 判定要素             | 最小構成が限界を迎える閾値    | 理由                                                           |
|:-----------------|:-----------------|:-------------------------------------------------------------|
| **画面数 / 機能数**    | **3 〜 5 画面**     | 画面間でのデータ共有や共通ロジックの抽出が必要になり、Fat UI ではコードの重複と管理不能なスパゲッティ化が始まる。 |
| **API エンドポイント数** | **5 個以上**        | 手書き DTO の管理コストと、型変更時の修正漏れ（人力テストの限界）が顕在化する。                   |
| **メンテナンス期間**     | **3 ヶ月以上**       | 過去に作った機能の「デグレード（先祖返り）」を確認する人力コストが、新規開発時間を上回り始める。             |
| **AI 活用範囲**      | **ロジックの 50% 以上** | Activity が 1000 行を超えると AI が全体の文脈を理解しきれず、不正確なコードを提案し始めるため。    |

### 結論：逆転のタイミング

本プロジェクトの規模（リポジトリ一覧 1 画面）であれば、最小構成の方が **約 2 〜 3 倍高速** に構築可能です。しかし、
**「リポジトリ詳細」「ユーザープロフィール」「お気に入り保存（ローカルDB）」「検索フィルタ」** の 4
つの機能を追加した時点で、本プロジェクトのアーキテクチャの方が **累計コストが低くなり、品質も安定する
** と推計されます。

特に AI エージェントを活用する場合、明確な設計図（レイヤー構造）があることで、AI
が「どこを修正すべきか」を迷わず、人間がコードレビューを行うコストも劇的に下がります。
