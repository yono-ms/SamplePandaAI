# AI 協調開発フロー (AI-Collaborative Development Flow)

このプロジェクトでは、AI エージェント（Gemini）の暴走を防ぎ、勘違いによる後戻りを最小限にするために、以下の厳格な開発フローを適用します。
各フェーズの終わりには「コミット」を挟み、AI は自律的な動作を停止してユーザーの指示を待ちます。

---

## コミットゲートと承認責任

各フェーズの境界には **「コミットゲート」** が設定されています。
コミットゲートにおける **承認者（ユーザー）は、その時点での変更内容（設計、コード、テスト結果等）の妥当性に対して最終的な責任を持つ
** ものとします。
AI は承認を得るために必要な情報（差分、テスト結果、エビデンス等）を過不足なく提示しなければなりません。

---

## フェーズ 1: 前準備 (Pre-preparation)

開発を開始する前に、AI は作業環境を整え、進捗を同期します。

- [ ] **プロジェクト構成の把握**: `docs/PROJECT_STRUCTURE.md` を読み込み、最新のディレクトリ構造とコンポーネント配置を正確に把握したか。
- [ ] **指針・ルールの同期**: `docs/AGENTS.md` および `docs/GEMINI.md` を読み込み、エージェントの役割と禁止事項を再認識したか。
- [ ] **ブランチ確認・提案**: `git status` で現状を確認し、タスクにふさわしいブランチ名を提案,
  または適切なブランチにいることを確認したか。
- [ ] **進捗認識の提示**: `docs/NEXT_STEPS.md` を確認し、現在の正確な進捗状況を提示したか。
- [ ] **開始許可の取得**: ユーザーから次フェーズ開始の明示的な許可を得たか。
- [ ] **NEXT_STEPS更新**: 許可を得た後、`NEXT_STEPS.md` のフェーズ情報を更新したか。
- [ ] **コミットメッセージの提案**: 承認時にコミットが必要な場合、適切なメッセージを提案したか。
- [ ] **【コミットゲート 1】**:
  - [ ] 準備作業に伴うドキュメント（`NEXT_STEPS.md`等）の更新のみが含まれているか。
  - [ ] 不要なコード修正が含まれていないか。

## フェーズ 2: 詳細設計 (Detailed Design)

タスクの概要に基づき、実装を開始する前に詳細な設計書（`docs/features/` 配下）を作成・確定させます。

- [ ] **既存コード調査**: 関連する既存コードを `grep` や `list_files` 等で調査し、整合性を確認したか。
- [ ] **ソースツリー更新案の反映**: 設計内容に従い、どのファイルを新規作成・修正・削除するか、具体的なパスを設計書に明記したか。
- [ ] **テスト観点の反映**: 実装すべきテストケースを具体的に設計書（`docs/features/` 配下）に明記し、必要に応じて
  `docs/TEST_STRATEGY.md` を更新したか。
- [ ] **設計書の作成/確定**: `docs/features/` 内に、ゴール、影響範囲、構成、テスト観点を網羅した設計書を作成・更新したか。
- [ ] **ユーザー承認**: ユーザーから設計書の内容に対する明示的な「承認」を得たか。
- [ ] **NEXT_STEPS更新**: 承認を得た後、`NEXT_STEPS.md` を更新したか。
- [ ] **コミットメッセージの提案**: 承認時に、適切なメッセージを提案したか。
- [ ] **【コミットゲート 2】**:
  - [ ] 差分が `docs/features/` 配下の設計ドキュメント、`docs/PROJECT_STRUCTURE.md`、および
    `docs/TEST_STRATEGY.md` の更新のみに限定されているか。
  - [ ] 設計書に「ソースツリー更新案」と「テスト観点」が具体的に記述されているか。
  - [ ] **承認責任**: ユーザーは、設計内容が要求を満たしていることを確認し、承認したか。

## フェーズ 3: 実装とビルド確認 (Implementation & Build)

確定した設計書に基づき、コードを記述します。

- [ ] **設計・指針の遵守**: フェーズ 2 で確定させたソースツリー構成、テスト観点、および指針から逸脱していないか。
- [ ] **並行実装**: アプリ本体のコードと、それに対応するテストコードをセットで実装したか。
- [ ] **ビルド確認**: `assembleDebug` および `assembleAndroidTest` 等が成功し、コンパイルエラーがないことを確認したか。
- [ ] **手戻り判断**: 実装中に設計の不備を発見した場合、独断で設計を変更せず、直ちに報告してフェーズ 2
  に戻る判断をしたか。
- [ ] **NEXT_STEPS更新**: 完了後、`NEXT_STEPS.md` を更新したか。
- [ ] **コミットメッセージの提案**: 承認時に、適切なメッセージを提案したか。
- [ ] **【コミットゲート 3】**:
  - [ ] 差分が設計書で指定されたファイル（ソースコードおよびテストコード）のみに限定されているか。
  - [ ] `assembleDebug` および `assembleAndroidTest` が成功し、コンパイルエラーがないか。
  - [ ] **承認責任**: ユーザーは、実装が設計通りであり、ビルドエラーがないことを確認し、承認したか。

## フェーズ 4: テストコード・レビュー (Test Code Review)

- [ ] **提示と説明**: 実装したコードとテストコードを提示し、設計書の「テスト観点」との対応関係を説明したか。
- [ ] **AI Silence**: **【重要】** コード提示後、ユーザーからのレビュー指摘があるまで、一切の追加修正や改善案の実装を行わずに待機したか。
- [ ] **指摘対応**: 指摘内容が設計変更を伴う場合、フェーズ 2 に戻って設計書を更新したか。
- [ ] **NEXT_STEPS更新**: 承認後、`NEXT_STEPS.md` を更新したか。
- [ ] **コミットメッセージの提案**: 承認時に、適切なメッセージを提案したか。
- [ ] **【コミットゲート 4】**:
  - [ ] レビュー指摘に対する修正が完了し、設計書との整合性が保たれているか。
  - [ ] **承認責任**: ユーザーは、コード品質およびテストコードの妥当性を確認し、承認したか。

## フェーズ 5: テスト実施 (Test Execution)

- [ ] **全テスト実行**: プロジェクトの全テスト（Unit Test / Instrumented Test）を実行したか。
- [ ] **失敗時の診断と報告**: テストが失敗した場合、原因を報告し、手戻り先（F2/F3）について
  **ユーザーの合意を得てから**修正したか。
- [ ] **PR説明文の提案**: テスト合格時、PRに記載すべき内容（概要、変更点、テスト結果）を Markdown
  形式で提案したか。
- [ ] **NEXT_STEPS更新**: テストパス後、または手戻り決定時に `NEXT_STEPS.md` を更新したか。
- [ ] **コミットメッセージの提案**: 承認時に、適切なメッセージを提案したか。
- [ ] **【コミットゲート 5】**:
  - [ ] 全テスト（Unit/Instrumented）がパスしているか。
  - [ ] **承認責任**: ユーザーは、テスト結果のエビデンスを確認し、PR 出しの準備が整ったことを承認したか。

## フェーズ 6: 外部レビュー対応 (External Review)

- [ ] **指摘の分析**: 外部（ユーザー等）からのレビュー指摘を分析し、修正の必要性を判断したか。
- [ ] **手戻り判定と承認**: 指摘対応が設計変更を伴うか判定し、手戻り先（F2/F3）についてユーザーの承認を得たか。
- [ ] **AI Silence**: レビュー指摘を待つ間、独断でコードを動かしていないか。
- [ ] **NEXT_STEPS更新**: レビュー通過、または手戻り決定時に `NEXT_STEPS.md` を更新したか。
- [ ] **コミットメッセージの提案**: 修正・承認時に、適切なメッセージを提案したか。
- [ ] **【コミットゲート 6】**:
  - [ ] レビュー指摘に対する修正が、承認された方針通りに行われているか。
  - [ ] **承認責任**: ユーザーは、指摘対応の内容および再実行したテスト結果を確認し、承認したか。

## フェーズ 7: ドキュメント最新化 (Final Documentation Sync)

- [ ] **ドキュメント同期**: `docs/PROJECT_STRUCTURE.md` および該当機能の設計書を、最終的な実装と完全に一致させたか。
- [ ] **NEXT_STEPS最終更新**: 全作業完了後、`NEXT_STEPS.md` を最終状態（完了）に更新したか。
- [ ] **コミットメッセージの提案**: 承認時に、適切なメッセージを提案したか。
- [ ] **【コミットゲート 7】**:
  - [ ] `docs/PROJECT_STRUCTURE.md` および設計書が最終実装と完全に一致しているか。
  - [ ] **承認責任**: ユーザーは、ドキュメントの最終状態を確認し、全作業の完了を承認したか。

---

## AI エージェントへの共通ルール

- **NEXT_STEPS.md の更新義務**: 各フェーズの完了報告を行い、ユーザーから許可を得た直後に、AI は必ず
  `docs/NEXT_STEPS.md` を最新の状態に更新しなければならない。
- **コミットメッセージの提案**: コミットを伴う全工程において、承認・完了時に AI
  は適切なコミットメッセージを提案しなければならない。
- **独断リファクタリングの禁止**: 設計書にない構造変更を行う場合は、必ず事前にフェーズ 2
  へ戻るプロセスを踏むこと。
- **セッション強制終了**: フェーズ完了報告（`--- PHASE [N] COMPLETE ---`
  ）を行った後は、即座にツール実行を停止し、ユーザーの指示を待つ。
- **実在性の再確認**: 実装中も、常に `find_files` や `read_file` を用いて、既存クラスやリソースの実在性を確認すること。
