# Gemini 活用詳細

このプロジェクトにおける Google Gemini (AI エージェント) の具体的な活用方法と、遵守すべき厳格なルールを定義します。

## Gemini への基本原則 (厳守事項)

**回答の正確性を担保するため、以下のルールを徹底してください。**

1. **実在性の確認**:
    - 回答に含めるすべてのクラス名、関数名、変数名は、**必ずプロジェクト内のファイルを検索して実在を確認してください。
      **
    - 記憶や推測に基づく「それらしい名前」の使用を禁止します。

2. **不明な場合の対応**:
    - 検索の結果、該当するクラス、関数、変数が確認できなかった場合は、*
      *推測せず、はっきりと「該当するクラス、関数、変数は存在しませんでした」と回答してください。**

3. **コード生成の基準**:
    - 既存のプロジェクト構造（パッケージ名、テーマ設定など）を尊重し、矛盾のないコードを生成してください。

## 厳格なフェーズ管理ルール (Hard Gate)

AI エージェントは、`docs/DEVELOPMENT_FLOW.md` に基づき、各フェーズの終わりにおいて以下のプロセスを遵守しなければなりません。

1. **フェーズ完了時の自動停止**:
    - 各フェーズ（設計、実装、テスト、レビュー等）の最後のアクションが完了した時点で、**AI
      は自律的な推論とツール実行を即座に停止しなければならない。**
    - 指示されていない追加の修正や、次のフェーズに向けた準備作業を独断で行うことを厳禁とする。

2. **完了報告と承認待ち**:
    - フェーズ終了時には、必ず以下の形式で報告を行い、**回答を終了せよ。**
   ```text
   --- PHASE [N] COMPLETE ---
   【完了事項】: (実施した内容)
   【現在の状態】: (ビルド成否、テスト結果等)
   【次フェーズ】: (次に進むべきフェーズ名)

   上記内容を確認し、問題なければコミットをお願いします。
   コミットメッセージ案: [type]: [description]
   ```

3. **セッション強制リセット (One-Phase, One-Chat)**:
    - **一つのチャットセッションで複数のフェーズを跨いで作業することを禁止する。**
    - フェーズが完了し、ユーザーによるコミットが完了した後は、必ずチャットを終了しなければならない。
   - 次のフェーズの指示は、必ず新しいチャットセッションで受け取ること。

4. **セッション開始時の同期と確認 (Session Start Protocol)**:
   - 新しいチャットセッションを開始した際、AI は以下の手順でプロジェクトの状態を把握すること。
       1. `docs/PROJECT_STRUCTURE.md` で構成を確認。
       2. `docs/NEXT_STEPS.md` で進捗と現在のターゲットタスクを確認。
       3. `git status` で現在のブランチと変更状況を確認。
   - 把握後、**即座に「実行系ツール（Build, Test, UI Render, Write）」を使用して作業を開始してはならない。
     **
   - AI はまずユーザーに対し、以下の情報を提示し、**次フェーズ開始の明示的な許可**を求めなければならない。
       - **現在の進捗状況の認識**（どのタスクのどのフェーズか）
       - **ブランチ状況の確認結果**（必要に応じて新規ブランチ名を提案）
       - **前フェーズのコミット完了の確認**
   - ユーザーからの明示的な合意を得るまでは、読み取り専用の調査（Search/Read）に留めること。

5. **AI Silence (待機状態) と解除条件**:
    - レビュー待ち等のフェーズにおいて、AI は自律的な提案を控え、ユーザーの応答を待たなければならない。
    - **解除条件**:
        - ユーザーからの明示的な指示（「continue」「続けて」等）があった場合。
        - ユーザーからのレビュー指摘、または「問題なし」の承認があった場合。
        - ユーザーの応答がないまま一定時間（または一定の対話ターン）が経過し、進捗確認のための再確認が許容される場合（ただし、独断でのコード修正は厳禁）。

## 共通ルール

- **NEXT_STEPS.md の更新義務**: 各フェーズの完了許可を得た直後、および作業完了報告の直前に、必ず
  `docs/NEXT_STEPS.md` を最新状態に更新すること。
- **コミットメッセージの提案**: 承認・完了時には、必ず適切なコミットメッセージ（および必要に応じて PR
  説明文）を提案すること。
- **フィードバックループ（手戻り）**: 実装中等に設計変更が必要になった場合、独断で進めず、原因に応じて
  F2 (設計) または F3 (実装) への手戻りを提案し、承認を得ること。
