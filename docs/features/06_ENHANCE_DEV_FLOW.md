# 開発プロセスの強化 (Enhancement of Development Flow)

## 1. ゴール (Goal)

`docs/DEVELOPMENT_FLOW.md` において、フェーズ番号を 1 から始まる体系 (1-7)
に振り直し、各フェーズに詳細な完了チェックリストを追加する。特に「構造の把握」と「ルールの遵守」を前準備に組み込み、詳細設計フェーズにおいて「ソースツリー構成」と「テスト観点」を具体的に設計書へ反映・確定させる。また、実装・テスト・レビュー工程におけるビルド確認の厳格化、手戻り承認フロー、および全工程でのコミットメッセージとPR説明文の提案を義務付ける。さらに、
**コミットゲートにおける承認責任を明確化**し、ユーザーが変更内容の妥当性に責任を持つ体制を確立する。
また、運用の柔軟性を高めるため、軽微な変更に対応した **「軽量フロー (Lite Flow)」** を導入する。
**進捗管理については、AIが外部ツール（GitHub
Issue等）を直接参照できない制約を考慮し、`docs/NEXT_STEPS.md` を「セッション間の唯一の同期手段」として再定義し、その更新を厳格に義務付ける。
**

## 2. 影響範囲 (Scope)

- `docs/DEVELOPMENT_FLOW.md`: 全フェーズの再定義 (1-7) とチェックリスト導入、承認責任の明記、および軽量フロー의
  定義追加。
  `NEXT_STEPS.md` による同期義務の再定義。
- `docs/GEMINI.md`: セッション開始プロトコルの具体化（ブランチ提案の追加）、およびフロー選択ルールの追加。
- `docs/NEXT_STEPS.md`: 進捗表示の更新ルールの明文化と記述の簡素化。

## 3. 追加・更新する内容

### フェーズ番号の再定義 (Standard Flow)

1. **フェーズ 1: 前準備 (Pre-preparation)**
2. **フェーズ 2: 詳細設計 (Detailed Design)**
3. **フェーズ 3: 実装とビルド確認 (Implementation & Build)**
4. **フェーズ 4: テストコード・レビュー (Test Code Review)**
5. **フェーズ 5: テスト実施 (Test Execution)**
6. **フェーズ 6: 外部レビュー対応 (External Review)**
7. **フェーズ 7: ドキュメント最新化 (Final Documentation Sync)**

### 軽量フロー (Lite Flow)

- **対象**: typo修正、ドキュメントの軽微な修正、小規模なバグ修正（影響範囲が閉じており、既存の設計変更を伴わないもの）。
- **適用フェーズ**:
  - **フェーズ 2 (設計)**: 簡略化した設計メモの作成と承認。
  - **フェーズ 3 (実装)**: 実装とビルド確認。
  - **フェーズ 5 (テスト)**: 関連するテストの実行と完了報告。
- **制限**: Lite Flow 適用中であっても、各フェーズの「コミットゲート」と「承認責任」は省略してはならない。また、途中で影響範囲が広いことが判明した場合は、直ちに
  Standard Flow (F2) へ移行すること。

### 共通ルール

- **NEXT_STEPS.md の更新義務（最重要）**: AI はセッション間のコンテキストを維持するため、各フェーズの完了報告時に必ず
  `docs/NEXT_STEPS.md` を最新の状態に更新しなければならない。AI は GitHub Issue
  を直接参照できないため、このファイルが「唯一の真実」となる。
- **コミットメッセージの提案**: コミットを伴う全工程において、承認・完了時に AI
  は適切なコミットメッセージを提案しなければならない。
- **フィードバックループ（手戻り）**: 工程進行中に設計変更や修正が必要になった場合、独断で進めず、原因（設計/実装/テスト等）に応じて
  F2 または F3 への手戻りをユーザーに提案し、承認を得ること。
- **承認責任の明確化**: 各フェーズの境界（コミットゲート）において、承認者（ユーザー）がその内容に対して最終責任を持つことを明文化する。
- **フローの選択**: フェーズ 1 において、タスクの規模に基づき Standard Flow または Lite Flow を提案すること。

### 強化されたチェックリスト

#### フェーズ 1: 前準備 (Pre-preparation)

AI は作業を開始する前に、以下の項目を実行・確認し、報告しなければならない。

- [ ] **プロジェクト構成の把握**: `docs/PROJECT_STRUCTURE.md` を読み込み、最新のディレクトリ構造とコンポーネント配置を正確に把握したか。
- [ ] **指針・ルールの同期**: `docs/AGENTS.md` および `docs/GEMINI.md` を読み込み、エージェントの役割と禁止事項を再認識したか。
- [ ] **ブランチ確認・提案**: `git status` で現状を確認し、タスクにふさわしいブランチ名を提案、または適切なブランチにいることを確認したか。
- [ ] **フローの選択提案**: タスクの規模に基づき、Standard Flow または Lite Flow の適用を提案したか。
- [ ] **進捗認識の提示**: **`docs/NEXT_STEPS.md` を確認し**、現在の正確な進捗状況を提示したか。
- [ ] **開始許可の取得**: ユーザーから次フェーズ開始の明示的な許可を得たか。
- [ ] **NEXT_STEPS更新**: 許可を得た後、`NEXT_STEPS.md` を更新したか。
- [ ] **コミットメッセージの提案**: 承認時にコミットが必要な場合、適切なメッセージを提案したか。

#### フェーズ 2: 詳細設計 (Detailed Design)

- [ ] **既存コード調査**: 関連する既存コードを `grep` 等で調査し、整合性を確認したか。
  - [ ] **特に、影響を受けるクラス、関数、およびその呼び出し元（Call Hierarchy）を網羅的に列挙したか。**
- [ ] **ソースツリー更新案の反映**: 設計内容に従い、どのファイルを新規作成・修正・削除するか、具体的なパスを設計書に明記したか。
- [ ] **テスト観点の反映**: 設計内容に従い、実装すべきテストケースを具体的に設計書に明記したか。
- [ ] **設計書の作成/確定**: `docs/features/` 内に、ゴール、影響範囲、構成、テスト観点等を網羅した設計書を作成・更新したか。
- [ ] **ユーザー承認**: ユーザーから設計書の内容に対する明示的な「承認」を得たか。
- [ ] **NEXT_STEPS更新**: 承認を得た後、`NEXT_STEPS.md` を更新したか。
- [ ] **コミットメッセージの提案**: 適切なメッセージを提案したか。
- [ ] **【コミットゲート 2】**: 確定した設計ドキュメントのみをコミットしたか。**承認責任**が確認されたか。

#### フェーズ 3: 実装とビルド確認 (Implementation & Build)

- [ ] **設計・指針の遵守**: フェーズ 2 で確定させた構成・観点から逸脱していないか。
- [ ] **ビルド確認**: `assembleDebug` および `assembleAndroidTest` が成功したか。
- [ ] **手戻り判断**: 設計変更が必要な場合、独断で進めずフェーズ 2 に戻る判断をしたか。
- [ ] **NEXT_STEPS更新**: 完了後、`NEXT_STEPS.md` を更新したか。
- [ ] **コミットメッセージの提案**: 適切なメッセージを提案したか。
- [ ] **【コミットゲート 3】**: 実装コードとテストコードをコミットしたか。**承認責任**が確認されたか。

#### フェーズ 4: テストコード・レビュー (Test Code Review)

- [ ] **提示と説明**: 実装したコードとテストコードを提示し、設計書の「テスト観点」との対応を説明したか。
- [ ] **AI Silence**: 指摘があるまで一切の追加修正・改善案の実装を行わずに待機したか。
- [ ] **指摘対応**: 指摘内容が設計変更を伴う場合、フェーズ 2 に戻って設計書を更新したか。
- [ ] **NEXT_STEPS更新**: 承認後、`NEXT_STEPS.md` を更新したか。
- [ ] **コミットメッセージの提案**: 適切なメッセージを提案したか。
- [ ] **【コミットゲート 4】**: レビュー完了後の状態をコミットしたか。**承認責任**が確認されたか。

#### フェーズ 5: テスト実施 (Test Execution)

- [ ] **全テスト実行**: Unit Test および Instrumented Test を実行したか。
- [ ] **失敗時の原因調査**: 失敗時、原因を調査し、手戻り先の承認を得たか。
- [ ] **PR説明文の提案**: テスト合格時、PRに記載すべき内容を提案したか。
- [ ] **NEXT_STEPS更新**: テストパス後、または手戻り決定時に `NEXT_STEPS.md` を更新したか。
- [ ] **コミットメッセージの提案**: 適切なメッセージを提案したか。
- [ ] **【コミットゲート 5】**: すべてのテストがパスした状態をコミットしたか。**承認責任**が確認されたか。

#### フェーズ 6: 外部レビュー対応 (External Review)

- [ ] **指摘の分析**: 外部レビュー指摘を分析し、修正の必要性を判断したか。
- [ ] **手戻り判定と承認**: 指摘対応が設計変更を伴うか判定し、ユーザーの承認を得たか。
- [ ] **AI Silence**: 指摘を待つ間、独断でコードを動かしていないか。
- [ ] **NEXT_STEPS更新**: レビュー通過、または手戻り決定時に `NEXT_STEPS.md` を更新したか。
- [ ] **コミットメッセージの提案**: 適切なメッセージを提案したか。
- [ ] **【コミットゲート 6】**: レビュー指摘対応後の状態をコミットしたか。**承認責任**が確認されたか。

#### フェーズ 7: ドキュメント最新化 (Final Documentation Sync)

- [ ] **ドキュメント同期**: `PROJECT_STRUCTURE.md` および該当機能の設計書を最終実装と完全に同期させたか。
- [ ] **NEXT_STEPS最終更新**: 全作業完了後、`NEXT_STEPS.md` を最終状態（完了）に更新したか。
- [ ] **コミットメッセージの提案**: 適切なメッセージを提案したか。
- [ ] **【コミットゲート 7】**: ドキュメント更新後の最終状態をコミットしたか。**承認責任**が確認されたか。

## 4. テスト観点 (Verification)

- **AIがGitHub Issueを読めない制約**を前提とした進捗同期フロー（`NEXT_STEPS.md` の義務化）が明文化されていること。
- 各フェーズのチェックリストが、これまでのレビュー指摘を正確に反映し、重複がないこと。
- **軽量フロー (Lite Flow)** が定義され、適用対象とフェーズが明記されていること。
- フェーズ 1 において、フローの選択提案が義務付けられていること。
- **コミットゲートにおける承認責任**が明文化されており、各チェックリストに含まれていること。
- **フェーズ 2 の既存コード調査**において、影響範囲（関連クラス・関数・呼び出し元）の網羅的列挙が義務付けられていること。
