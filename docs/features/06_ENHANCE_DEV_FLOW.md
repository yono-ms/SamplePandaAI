# 開発プロセスの強化 (Enhancement of Development Flow)

## 1. ゴール (Goal)

`docs/DEVELOPMENT_FLOW.md` において、フェーズ番号を 1 から始まる体系 (1-7)
に振り直し、各フェーズに詳細な完了チェックリストを追加する。特に「構造の把握」と「ルールの遵守」を前準備に組み込み、詳細設計フェーズにおいて「ソースツリー構成」と「テスト観点」を具体的に設計書へ反映・確定させる。また、実装・テスト・レビュー工程におけるビルド確認の厳格化、手戻り承認フロー、および全工程でのコミットメッセージとPR説明文の提案を義務付ける。

## 2. 影響範囲 (Scope)

- `docs/DEVELOPMENT_FLOW.md`: 全フェーズの再定義 (1-7) とチェックリスト導入
- `docs/GEMINI.md`: セッション開始プロトコルの具体化（ブランチ提案の追加）
- `docs/NEXT_STEPS.md`: 進捗表示の更新ルールの明文化

## 3. 追加・更新する内容

### フェーズ番号の再定義

1. **フェーズ 1: 前準備 (Pre-preparation)**
2. **フェーズ 2: 詳細設計 (Detailed Design)**
3. **フェーズ 3: 実装とビルド確認 (Implementation & Build)**
4. **フェーズ 4: テストコード・レビュー (Test Code Review)**
5. **フェーズ 5: テスト実施 (Test Execution)**
6. **フェーズ 6: 外部レビュー対応 (External Review)**
7. **フェーズ 7: ドキュメント最新化 (Final Documentation Sync)**

### 共通ルール

- **NEXT_STEPS.md の更新義務**: 各フェーズの完了報告を行い、ユーザーから許可を得た直後に、AI は必ず
  `docs/NEXT_STEPS.md` を最新の状態に更新しなければならない。
- **コミットメッセージ host 提案**: コミットを伴う全工程において、承認・完了時に AI
  は適切なコミットメッセージを提案しなければならない。
- **フィードバックループ（手戻り）**: 工程進行中に設計変更や修正が必要になった場合、独断で進めず、原因（設計/実装/テスト等）に応じて
  F2 または F3 への手戻りをユーザーに提案し、承認を得ること。

### 強化されたチェックリスト

#### フェーズ 1: 前準備 (Pre-preparation)

AI は作業を開始する前に、以下の項目を実行・確認し、報告しなければならない。

- [ ] **プロジェクト構成の把握**: `docs/PROJECT_STRUCTURE.md` を読み込み、最新のディレクトリ構造とコンポーネント配置を正確に把握したか。
- [ ] **指針・ルールの同期**: `docs/AGENTS.md` および `docs/GEMINI.md` を読み込み、エージェントの役割と禁止事項を再認識したか。
- [ ] **ブランチ確認・提案**: `git status` で現状を確認し、タスクにふさわしいブランチ名を提案、または適切なブランチにいることを確認したか。
- [ ] **進捗認識の提示**: `docs/NEXT_STEPS.md` を確認し、現在の正確な進捗状況を提示したか。
- [ ] **開始許可の取得**: ユーザーから次フェーズ開始の明示的な許可を得たか。
- [ ] **NEXT_STEPS更新**: 許可を得た後、`NEXT_STEPS.md` のフェーズ情報を更新したか。
- [ ] **コミットメッセージの提案**: 承認時にコミットが必要な場合、適切なメッセージを提案したか。

#### フェーズ 2: 詳細設計 (Detailed Design)

このフェーズの主目的は、**設計書（docs/features/ 配下）の作成・更新**を通じて、実装内容を完全に確定させることである。

- [ ] **既存コード調査**: 関連する既存コードを `grep` 等で調査し、整合性を確認したか。
- [ ] **ソースツリー更新案の反映**: 設計内容（ゴール、影響範囲等）に従い、どのファイルを新規作成・修正・削除するか、具体的なパスを設計書に明記したか。
- [ ] **テスト観点の反映**: 設計内容（ゴール、影響範囲等）に従い、実装すべきテストケースを具体的に設計書に明記したか。
- [ ] **設計書の作成/確定**: `docs/features/` 内に、上記（ゴール、影響範囲、構成、テスト観点等）を網羅した設計書を作成、または既存のものを更新したか。
- [ ] **ユーザー承認**: ユーザーから設計書の内容（ソースツリーおよびテスト観点の更新内容を含む）に対する明示的な「承認」を得たか。
- [ ] **NEXT_STEPS更新**: 承認を得た後、`NEXT_STEPS.md` を更新したか。
- [ ] **コミットメッセージの提案**: 【コミットゲート 2】に向けて、適切なメッセージを提案したか。
- [ ] **【コミットゲート 2】**: 確定した設計ドキュメントのみをコミットしたか。

#### フェーズ 3: 実装とビルド確認 (Implementation & Build)

- [ ] **設計・指針の遵守**: フェーズ 2 で確定させたソースツリー構成およびテスト観点、ならびに
  `AGENTS.md` / `GEMINI.md` の指針から逸脱していないか。
- [ ] **ビルド確認**: `assembleDebug` および `assembleAndroidTest` が成功したか。
- [ ] **手戻り判断**: 設計変更が必要な場合、独断で進めずフェーズ 2 に戻る判断をしたか。
- [ ] **NEXT_STEPS更新**: 許可を得た後、`NEXT_STEPS.md` を更新したか。
- [ ] **コミットメッセージの提案**: 【コミットゲート 3】に向けて、適切なメッセージを提案したか。
- [ ] **【コミットゲート 3】**: 実装コードとテストコードをコミットしたか。

#### フェーズ 4: テストコード・レビュー (Test Code Review)

- [ ] **提示と説明**: 実装したコードとテストコードを提示し、設計書の「テスト観点」との対応関係を説明したか。
- [ ] **AI Silence**: 指摘があるまで一切の追加修正・改善案の実装を行わずに待機したか。
- [ ] **指摘対応**: 指摘内容が設計変更を伴う場合、フェーズ 2 に戻って設計書を更新したか。
- [ ] **NEXT_STEPS更新**: 承認後、`NEXT_STEPS.md` を更新したか。
- [ ] **コミットメッセージの提案**: 【コミットゲート 4】に向けて、適切なメッセージを提案したか。
- [ ] **【コミットゲート 4】**: レビュー完了後の状態をコミットしたか。

#### フェーズ 5: テスト実施 (Test Execution)

- [ ] **全テスト実行**: Unit Test および Instrumented Test を実行したか。
- [ ] **合否判定と報告**: テスト結果を正確に報告したか。
- [ ] **失敗時の原因調査**: 失敗時、原因（本体コード/設計書/テストコード）を調査し、手戻り先（F2 or F3）の
  **承認を得たか**。
- [ ] **PR説明文の提案**: テスト合格時、PRに記載すべき内容（概要、変更点、テスト結果）を Markdown
  形式で提案したか。
- [ ] **NEXT_STEPS更新**: テストパス後、または手戻り決定時に `NEXT_STEPS.md` を更新したか。
- [ ] **コミットメッセージの提案**: 【コミットゲート 5】に向けて、適切なメッセージを提案したか。
- [ ] **【コミットゲート 5】**: すべてのテストがパスした状態をコミットしたか。

#### フェーズ 6: 外部レビュー対応 (External Review)

- [ ] **指摘の分析**: 外部（ユーザー等）からのレビュー指摘を分析し、修正の必要性を判断したか。
- [ ] **手戻り判定と承認**: 指摘対応が設計変更を伴うか判定し、手戻り先（F2/F3）についてユーザーの承認を得たか。
- [ ] **AI Silence**: レビュー指摘を待つ間、独断でコードを動かしていないか。
- [ ] **NEXT_STEPS更新**: レビュー通過、または手戻り決定時に `NEXT_STEPS.md` を更新したか。
- [ ] **コミットメッセージの提案**: 修正・承認時に、適切なメッセージを提案したか。
- [ ] **【コミットゲート 6】**: レビュー指摘対応後の状態をコミットしたか。

#### フェーズ 7: ドキュメント最新化 (Final Documentation Sync)

- [ ] **ドキュメント同期**: `PROJECT_STRUCTURE.md` および該当機能の設計書を最終実装と完全に同期させたか。
- [ ] **NEXT_STEPS最終更新**: 全作業完了後、`NEXT_STEPS.md` を最終状態（完了）に更新したか。
- [ ] **コミットメッセージの提案**: 【コミットゲート 7】に向けて、適切なメッセージを提案したか。
- [ ] **【コミットゲート 7】**: ドキュメント更新後の最終状態をコミットしたか。

## 4. テスト観点 (Verification)

- 各フェーズのチェックリストが、これまでのレビュー指摘を正確に反映し、重複がないこと。
- フェーズ 6 (外部レビュー) において、手戻り判定と承認フローが義務付けられていること。
- フェーズ 7 (ドキュメント) が、実装完了後の最終同期工程として定義されていること。
- 全てのフェーズにおいて、承認・更新・提案のセットが網羅されていること。
